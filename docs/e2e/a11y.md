# Accessibility (axe-core) Testing Guide

## Overview

STOA's E2E suite gates PRs on WCAG 2.1 AA compliance using axe-core via `@axe-core/playwright`.

**Gate policy**

| Impact level | CI behaviour |
|---|---|
| `critical` | Hard fail (test throws) |
| `serious` | Hard fail (test throws) |
| `moderate` | Warn-only (annotation, no fail) |
| `minor` | Warn-only (annotation, no fail) |

The threshold is configurable per-call (`level` param to `assertNoA11yViolations`), and overridable globally via `A11Y_IMPACT_THRESHOLD` env var.

---

## Using the helpers

### `assertNoA11yViolations` — recommended

```typescript
import { assertNoA11yViolations } from '../../fixtures/a11y';

test('my page has no critical/serious violations', async ({ page }, testInfo) => {
  await page.goto('/my-route');
  await page.waitForLoadState('networkidle');

  await assertNoA11yViolations(page, testInfo);
});
```

The function:
1. Runs axe-core with `wcag2a` + `wcag2aa` tags.
2. Throws if any violations at `>= serious` are found (default).
3. Adds `a11y-warning` annotations for moderate/minor.
4. Attaches `a11y-violations.json` to the test report (always).

### `auditA11y` — for custom logic

```typescript
import { auditA11y } from '../../fixtures/a11y';

const result = await auditA11y(page, { tags: ['wcag2a', 'wcag2aa', 'wcag21aa'] });
console.log(result.violations); // critical + serious only
console.log(result.all);        // all severities
```

## Allowlist (known violations)

Use the `exclude` option to skip third-party or in-progress components:

```typescript
await assertNoA11yViolations(page, testInfo, 'serious', {
  exclude: [
    '#keycloak-iframe',          // third-party auth widget
    '[data-testid="legacy-chart"]', // known violation, tracked in CAB-XXXX
  ],
});
```

Keep a comment with the tracking ticket so the exclusion gets removed when fixed.

## Adding new Console pages

Edit `e2e/tests/a11y/console-dashboard-a11y.spec.ts` and add an entry to `CONSOLE_PAGES`:

```typescript
const CONSOLE_PAGES = [
  // ... existing entries ...
  { name: 'My New Page', path: '/my-new-route' },
];
```

The test loop generates one test per entry automatically.

## Running locally

Requires a built Console (no live backend needed — mock routes are injected):

```bash
# Build Console first
cd control-plane-ui && npm run build && cd ..

# Run only a11y project
cd e2e && npx playwright test --project=a11y

# Or the smoke-mock a11y scan (original gate)
cd e2e && npx playwright test --config=smoke-mock/playwright.config.ts a11y-axe
```

## CI wiring

Workflow: `.github/workflows/e2e-a11y-gate.yml`

Triggers on PRs that touch:
- `control-plane-ui/**`
- `e2e/smoke-mock/**`
- `e2e/fixtures/a11y.ts`
- `e2e/tests/a11y/**`

Steps:
1. Build Console (no live backend)
2. Run smoke-mock axe scan (`e2e/smoke-mock`)
3. Run `a11y` Playwright project (`e2e/tests/a11y/`)
4. Upload `a11y-violations-<run-id>` artifact (14-day retention)

## WCAG 2.1 AA Conformance Statement

STOA targets **WCAG 2.1 Level AA** conformance for the Console and Portal UIs.

Current automated coverage:
- All routes in `CONSOLE_PAGES` (10+ pages via smoke-mock, 5 via `tests/a11y/`)
- Scanned with `wcag2a`, `wcag2aa` axe-core rule sets

Gaps (manual testing recommended):
- Keyboard-only navigation flows
- Screen reader compatibility (NVDA, VoiceOver)
- Color contrast in dark/light theme switching

Known exclusions are tracked per-spec with a Linear ticket reference. Any exclusion without a ticket is a CI warning treated as a violation in the next ratchet cycle.

**Ratchet schedule** (per `axe-helper.ts` comment):
- Launch: block on `critical` only
- +2 cycles: raise threshold to `serious`
- +4 cycles: raise threshold to `moderate`
